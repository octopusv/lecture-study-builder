# プロジェクト概要

このワークスペースは、大学の授業動画から以下を作成する学習ポータルである。

- 全授業の高精度な文字起こし
- スライドのフレーム抽出とOCR
- 可能な限り詳細な授業内容Markdown
- 試験に出そうな4択問題
- 暗記カード
- 教科別のレスポンシブWebサイト
- ルートの教科一覧学習ポータル

ユーザー向けの文章、進捗報告、生成資料は日本語で書くこと。短い状態報告・実況、ツール実行前後の説明、中間経過、最終報告も含め、ユーザーに表示される地の文は常に日本語にする。英語に流れないこと（コード・コマンド・ファイルパス・識別子はそのままでよい）。
コミットを求められた場合、コミットメッセージは日本語にすること。

# 実装方針

公開テンプレートをcloneした直後は、教科データや完成済み成果物が存在しない前提で動作させる。
既存教科がある場合は、その成果物、サイト、台帳を先に確認し、正常な出力を壊さず未完了工程から再開する。
共通処理はルートの `scripts/`、プロジェクトSkill、`portal.css` を優先して再利用・汎用化する。
特定の大学、学部、教科名、講義回数をコードへ固定しない。
ルートの `index.html` はローカル生成物としてGit管理外にする。clone直後などファイルが存在しない場合は、`/setup` が `.claude/skills/setup/assets/index.html` から汎用初期ページを生成する。既存の `index.html` は個人用ホームを含めて絶対に上書きしない。

# 教科台帳

対象教科は `config/subjects.json` に登録された内容を唯一の基準とする。公開テンプレートには空の `config/subjects.example.json` を含め、利用者は `/setup-course <教科ID> <教科名>` で任意の教科を登録する。標準構成は15回×各6動画、命名は `1_1.mp4` から `15_6.mp4` だが、異なる場合は台帳の `lecture_count`、`parts_per_lecture`、`expected_videos` を変更する。
大容量動画を移動、複製、再エンコードしないこと。既存ファイルを削除しないこと。

# 初回セットアップ

`教科別/`とローカル教科台帳を含めずに配布されたリポジトリをcloneした直後は、Claude Codeで `/setup` を実行する。Skillは空の `config/subjects.json` と `教科別/` を生成し、ffmpeg・uv・Node、Python仮想環境、Apple Vision OCR依存、Whisperモデルを再実行可能な形で準備・検証する。Git初期化、`.gitignore`、Git履歴、動画ファイルは変更しない。環境構築後は `/setup-course 01 教科名` の形式で任意の教科を追加する。

# 標準実行方法

教科分析は、Claude Codeを再起動してプロジェクトSkillを読み込ませた後、原則として `/analyze-course <教科ID>` で開始する。例: `/analyze-course 04`。Skillは教科一覧から対象を解決し、既存成果物を検査して未完了工程から再開する。Skill内のStopフックが完成条件を検証し、未完了なら自動継続する。Claude Codeの保護機構でStopフックの連続継続は最大8回なので、上限に達した場合は同じコマンドを再実行する。

# 自律作業の原則

- 調査だけで止まらず、実装、長時間処理、生成、検証まで完走する。
- 資料作成、講義回ごとの分析、問題生成、品質監査、UI検証などに、必要なだけサブエージェントを使用してよい。独立した作業は積極的に並列化し、各サブエージェントの担当範囲と出力先を明確にして、重複作業やファイル競合を避けること。
- サブエージェントへ任せた成果はそのまま採用せず、主担当が内容の正確性、重複、欠損、選択肢品質、ファイル整合性を最終確認して統合すること。
- 問題・誤答選択肢・解説・暗記カード本文の作成には、プロジェクト定義の `lecture-assessment-author` サブエージェントを必ず使用する。このエージェントは `model: sonnet` で、利用可能な最新Sonnetを使う。
- 作成後は別の `assessment-quality-auditor` Sonnetサブエージェントで講義資料との照合と品質監査を必ず行う。同じエージェントに自己採点だけをさせて完了しない。
- 全講義回の授業内容まとめの統合編集、回間の矛盾・重複・欠落の最終監査、試験重要度の最終判断には、プロジェクト定義の `course-integration-editor` サブエージェントを必ず使用する。このエージェントは `model: opus` で、利用可能な最新Opusを使う。問題・暗記カードもSonnetによる回別監査後、別起動の同Opusエージェントが全講義横断の重複、範囲偏り、重要論点の出題漏れを最終監査する。
- 通常の実装判断は自分で行い、ユーザーへの質問は本当に進行不能な場合だけにする。
- 長時間コマンドが動いている間は適切に待機・監視し、必要な処理を途中で放置しない。
- 各工程は再実行可能にする。正常な既存成果物はスキップし、欠損・空ファイルだけを再処理する。
- 失敗した動画を記録し、再試行して最終的な欠損をゼロにする。
- 音声認識やOCRの原文を根拠にする。OCRの明らかな誤字は文脈から補正してよいが、講義にない内容を講義内容として捏造しない。
- 一般知識を補う場合は、講義範囲から自然に導ける標準的な定義に限定する。
- 数式、数値、固有名詞は文字起こしだけで決めず、OCRと前後文脈も照合する。

# 推奨処理方法

## 既存のPython環境とスクリプト

新しい処理系をゼロから作る前に、ルートの既存Python環境とスクリプトを確認して再利用・汎用化すること。

- Python仮想環境: `.venv/`
- 音声認識: `scripts/transcribe_macro.py`
- スライドOCR: `scripts/ocr_slides.py`
- 詳細Markdown生成: `scripts/build_detailed_notes.py`

これらは既存教科の処理に使用する共通基盤である。教科名、入力動画ディレクトリ、出力先、Whisperの初期プロンプトを引数で受け取り、任意の教科に安全に適用できる状態を維持する。
同じ機能を持つ別スクリプトを不必要に重複作成しないこと。

## 音声認識

- 既存 `.venv` と `mlx_whisper` を優先して再利用する。
- 基準モデルは `mlx-community/whisper-large-v3-turbo`。
- 言語は日本語、temperatureは0。
- 教科ごとに講義分野へ合わせた `initial_prompt` を使う。
- JSON、TXT、SRTなど、後工程と人間の確認に必要な形式を保存する。

## スライドOCR

- `ffmpeg` で30秒ごとを目安にフレーム抽出する。
- OCR処理はPythonスクリプトからmacOS標準のApple Vision OCRを呼び出す。Python独自OCRや外部OCR APIではない。
- 既存の `scripts/ocr_slides.py` はPyObjC経由で `Foundation` と `Vision` を使用し、`VNRecognizeTextRequest` を実行する実装である。まずこれを再利用する。
- 認識言語は日本語 `ja-JP` と英語 `en-US`、高精度認識と言語補正を使用する。
- 動画ごとに秒数、タイムスタンプ、OCR本文をJSONへ保存する。
- フレーム抽出完了を示すマーカーを用意し、再実行時に安全にスキップする。

## Markdown

各教科に最低限、以下を作る。

- `output/<教科名>_講義全記録.md`
- `output/<教科名>_授業内容まとめ_超詳細.md`
- `output/<教科名>_講義トピック一覧.md`
- `output/notes/第01回_超詳細.md` から教科台帳の最終講義回まで
- `output/transcripts/` の90動画分
- `output/slide_ocr/` の90動画分
- `output/README.md`

`<教科名>_講義全記録.md` は、15個の回別詳細ノートを時系列に結合した参照用アーカイブである。スライド記載事項と講師の説明を分離し、動画番号と `[MM:SS]` を付ける。このファイルは機械的な連結でもよいが、「授業内容まとめ」と呼ばないこと。

`<教科名>_授業内容まとめ_超詳細.md` は、単なる文字起こし・OCR・回別ノートの連結ではない。Claudeのサブエージェントが全講義回の詳細ノートを回ごとに読み、内容を理解して自分の言葉で再構成した、学習用の超詳細な統合解説書とする。最低限、以下を含める。

- 各回の目的、中心テーマ、重要概念、人物・理論・制度、具体例
- 定義、因果関係、比較、論争点、前提条件、例外、図表・モデルの読み方
- 講師が強調・反復した点と、試験で問われやすい理由
- 各回内部の論理展開と、前後の回とのつながり
- 全講義回を横断する主要テーマ、概念間の関係、全体のストーリー
- 混同しやすい概念、誤解しやすい点、覚えるべき固有名詞・年代・公式
- 各節の根拠となる講義回、動画番号・時刻または回別ノートの見出し

まとめ本文は、OCR全文や文字起こしを大量に貼り付けず、講義内容を省略しすぎない詳細な解説文として執筆する。資料を読んでいない別のAIや学生が、このファイルだけでも授業の論理と重要事項を理解できる密度を目指す。

全講義回分の文字起こし・OCR・詳細ノートを一つのサブエージェントへ無条件に一括投入しない。各回の詳細な再構成を1回単位のSonnetサブエージェントへ分担し、その成果を `course-integration-editor` Opusサブエージェントが全体構成に沿って統合編集する。統合後は、統合執筆とは別起動の同Opusエージェントが回間の重複、矛盾、欠落、OCR誤認、試験重要度を横断監査する。Opusは回別再構成MD、トピック一覧、根拠索引を主入力とし、必要な箇所だけ該当回の原資料へ戻る。

# Webサイトの品質基準

各教科の `site/` は、以下の機能と品質基準を満たす。

UIは絶対に妥協せず、完成品として洗練させること。単なる素のHTML、仮置き画面、マクロ経済サイトの文字と色だけを変えたコピーでは完了としない。各教科の性格に合った独自のビジュアル方向性を設計しつつ、学習効率を最優先する。

- 明確な視覚階層、十分な余白、読みやすい日本語組版、一貫した色・タイポグラフィ・コンポーネント
- 問題文、選択肢、正誤、解説、進捗、フィルターが一目で区別できる情報設計
- 数百問規模を扱っても操作に迷わず、重く見えず、長時間学習しても疲れにくい画面
- hover、focus、active、disabled、正解、不正解、学習済みなど全状態を丁寧に設計
- キーボード操作、フォーカス表示、コントラスト、適切なARIAなど基本的なアクセシビリティ
- PCとスマートフォンの両方で、横切れ、文字の重なり、押しにくいボタン、不自然な余白を残さない
- 装飾過多や派手なアニメーションより、速さ、可読性、理解しやすさを優先
- ブラウザでスクリーンショットを確認し、見た目の問題を修正してから完了する

- 全講義回の概要
- 暗記必須事項
- 4択問題
- 暗記カード
- キーワード検索
- 授業回と重要度による絞り込み
- 40問程度ごとのページ分割
- ランダム問題
- 未回答問題だけの表示
- 即時採点、正解、詳しい解説
- 詳細ノートまたは元動画の該当位置へのリンク
- 学習進捗と回答履歴のlocalStorage保存
- 用語→説明、説明→用語の双方向暗記カード
- PCと幅390px程度のスマートフォンで崩れないレスポンシブ表示

問題はすべて4択とし、各問の選択肢は4個かつ同一問題内で重複なしにする。
正解が一意に決まらない問題、意味のない選択肢、表現だけを変えた水増しを避ける。
講義内で明示された問題、講師の強調、まとめ、定義、人物、公式、因果関係、比較、計算、図表の読み方を優先する。

1教科につき、重複を除いた重要な知識項目を可能な限り収録する。量の目安は以下とする。

- 4択問題: 1講義回あたり20〜30問。全体目安は `lecture_count × 20〜30`
- 暗記カード: 1講義回あたり15〜25枚。全体目安は `lecture_count × 15〜25`。内容密度が特に高い回のみ30枚程度まで可

上記は最低数を強制するノルマではない。講義内容の密度に応じて各回の数を調整し、重要論点を重複なく網羅した時点で作問を止める。数合わせの言い換え、些末な事項の過剰出題、根拠の弱い問題、同じ知識を問うだけの水増しは禁止する。問題数より、講義範囲の網羅性、試験への有用性、正解の一意性、誤答選択肢と解説の品質を優先する。

必要な場合は、用語→説明、説明→用語、因果、比較、正誤、計算など異なる想起方向を使う。ただし、想起方向を変えただけの項目を無条件に別問題として数えない。

## 問題・暗記カードの作成手順

問題と暗記文言は、単語リストからJavaScriptやPythonで機械的に量産して完了してはならない。

1. 先に全入力動画の文字起こし、OCR、15個の回別詳細ノート、講義全記録、AIが再構成した授業内容まとめ超詳細版を完成させる。
2. 全講義回を複数の `lecture-assessment-author` Sonnetサブエージェントへ分担する。原則として1エージェントに割り当てるのは最大1講義回だけとし、複数回の詳細MDをまとめて読ませない。各担当は該当回の詳細MDを実際に読んで、問題、4選択肢、正解、解説、暗記カードを執筆する。
3. 1講義回の詳細MDが30万文字を超える、コンテキスト残量警告が出る、または出力が途中で切れる場合は、その回を動画1〜3と動画4〜6などの節単位へさらに分割する。分割担当の成果は別の統合担当が重複を除いてまとめる。
4. 各項目に、根拠となる講義回、動画番号、時刻または詳細ノートの節を保持する。
5. 別の `assessment-quality-auditor` Sonnetサブエージェントへ分担し、正解の一意性、誤答の妥当性、講義との整合、重複、網羅性を監査・修正する。監査担当も原則1講義回単位とする。
6. 回別監査後、別起動の `course-integration-editor` Opusサブエージェントが全講義回を横断し、問題・カードの重複、範囲偏り、重要論点の出題漏れ、試験重要度を最終監査する。
7. 主担当が全回を統合し、構文検査と重複検査を行い、Webサイトへ組み込む。

定型変換を補助的に使うことはできるが、それだけで作った項目を未確認で公開データへ入れないこと。最終的な問題・カードはすべてSonnetによる執筆または内容確認と、監査担当による品質監査を通すこと。目安の上限へ到達するためだけの水増しは行わない。

# ポータル更新

教科サイト完成後は、既存のルート `index.html` を維持しつつ対象教科のカードだけを更新する。`index.html` が存在しない場合は、先に `/setup` と同じ汎用初期ページ生成処理を実行する。

- 対象教科を「利用可能」にする
- 実際の問題数、カード数、動画数を表示する
- カードから対象教科のサイトへ正しく移動できるようにする
- 他教科のカード、集計値、リンクを壊さない
- 対象外の見出し、説明、デザイン、カードを汎用ページで置き換えない

# 公開版・非公開版の分離

クラウド公開を行う場合、ワークスペース全体や既存教科フォルダをそのままデプロイしてはならない。必ず生成物を次の3層へ分離する。

## 1. 公開版

`deploy/public/` に、一般公開してよいファイルだけを明示的な許可リスト方式で生成する。

- 公開可能: トップページ、教科サイト、独自に再構成した4択問題、暗記カード、短い自作要点、CSS、JavaScript
- 公開禁止: `videos/`、`output/`、文字起こし全文、OCR全文、スライド画像、超詳細MD、元動画リンク、動画ファイル名、動画時刻、内部ファイルパス、`.venv/`、`scripts/`、`.claude/`、`CLAUDE.md`
- 公開データから `video`、`seconds`、`sourceNote` など内部根拠フィールドを削除する
- 「動画を開く」「詳細ノート」など非公開資料へのリンクを表示しない
- 元講義の文章を長くコピーせず、問題、選択肢、解説、暗記文言を独自表現にする

除外リストだけに頼らず、公開可能なファイルだけを新規ディレクトリへコピー・生成する。

## 2. 非公開クラウド版

`deploy/private/` に、自分専用の詳細な学習サイトを生成する。

- Cloudflare Pagesの公開版とは別プロジェクトまたは別ホスト名にする
- Cloudflare Accessを前段に置き、指定した本人のメールアドレスまたはCloudflareアカウントだけをAllowする
- 認証、パスワード、管理者アカウント、Cookieをアプリ内で自作しない
- セッションCookieとログイン画面はCloudflare Accessへ任せる
- Access未設定の状態で非公開版をデプロイしない。可能ならdeny-by-defaultを使う
- `*.pages.dev`など認証を迂回できる別URLが残らないよう確認する
- 動画は含めない。生のOCR全文と文字起こし全文も初期設定では含めない
- 詳細な自作ノート、根拠表示、学習用メタデータはAccess保護下でのみ利用可能にしてよい

## 3. ローカル完全版

既存ワークスペースのサイトをローカル完全版として維持する。

- 動画、OCR、文字起こし、超詳細MDを閲覧できるのはローカル完全版だけ
- ローカルHTTPサーバーは `127.0.0.1` のみにbindする
- 公開版・非公開クラウド版の生成処理で元ファイルを移動・削除しない

## 公開前検査

デプロイ用ディレクトリを機械検査するスクリプトを作る。

- 動画・音声・画像フレームが0件
- 1ファイルも25MiBを超えない
- 禁止ディレクトリ名、内部絶対パス、`../videos/`、`../output/`、`.mp4`リンクが0件
- 公開版には transcript、OCR、超詳細ノート本文、動画番号・時刻などが0件
- 公開版と非公開版のファイル一覧をレポート出力する
- 検査に失敗した場合はデプロイを中止する

# 検証と完了報告

完了前に、コマンド出力として少なくとも以下を確認する。

- 対象動画数が教科台帳の `expected_videos` と一致
- 空でない文字起こしJSON数が `expected_videos` と一致
- 空でないOCR JSON数が `expected_videos` と一致
- 分割詳細ノート数が `lecture_count` と一致
